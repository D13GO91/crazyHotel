🌐 Visão Geral da Arquitetura

O Crazy Hotel é uma releitura digital do jogo de dedução social "The Resistance" (ou Avalon). O jogo funciona no modelo de TV-Controller:

  • Dono/TV (Host): Cria a sala e exibe um painel geral na TV ou projetor. A TV exibe o andamento do jogo, placares, votações abertas e o histórico de missões de forma passiva.
  • Jogadores (Clients/Players): Entram no jogo pelo celular usando o código de 4 letras. Cada um vê na sua tela informações confidenciais (como seu papel secreto) e executa ações ativas (propor equipes, votar ou assassinar).
🚀 Por que essa arquitetura é moderna?
Esta aplicação é 100% Serverless (sem servidor tradicional). Toda a comunicação em tempo real, persistência de banco e segurança de regras de negócio é feita combinando o React (Frontend) diretamente com os recursos integrados do Supabase (Backend as a Service) e PostgreSQL.

💻 O Ecossistema Frontend

Construído sobre o React 19 (a versão mais recente do framework) e estruturado com TypeScript para garantir segurança de tipos durante a escrita de código. O empacotador é o Vite 8, garantindo tempos de carregamento instantâneos no desenvolvimento local e builds enxutos em produção.

⚡ A Engrenagem Backend

O Supabase foi adotado para evitar a necessidade de criar e gerenciar um servidor Node.js/Python complexo. Ele expõe uma API REST automática direto no banco PostgreSQL, autenticação nativa e o mecanismo de Realtime baseado em WebSockets para escuta de notificações de novos registros na base de dados.

🗄️ Banco de Dados & Segurança Avançada

Para um desenvolvedor júnior, o maior erro comum é programar toda a lógica do jogo no frontend. Por exemplo: se o jogador quer ver seu papel, o frontend consulta todos os papéis e filtra o dele.

⚠️ Risco de Trapaça: Se os papéis secretos fossem enviados para o frontend direto em uma tabela pública, qualquer jogador mais técnico poderia simplesmente abrir o DevTools do Chrome (F12), olhar as variáveis de estado do React ou a aba Network, e descobrir o papel secreto de todos antes do início do jogo!

O Padrão de Segurança do Crazy Hotel

Para resolver isso, o Crazy Hotel utiliza o conceito de Tabelas Privadas no Supabase combinadas com Security Definer RPCs (Remote Procedure Calls).

SQL schema.sql 
-- Tabela Pública (Sem papéis secretos, apenas status)
create table players (
  id uuid primary key default gen_random_uuid(),
  room_id uuid not null references rooms(id) on delete cascade,
  name varchar(100) not null,
  is_host boolean not null default false,
  team_vote varchar(50), -- Voto público de proposta de time
  has_voted_mission boolean not null default false, -- Apenas indica se votou
  status varchar(50) not null default 'CONNECTED'
);

-- Tabela Privada (Excluída das políticas de leitura pública!)
create table player_roles (
  player_id uuid primary key references players(id) on delete cascade,
  role varchar(50) not null -- 'GUEST', 'THIEF', 'MANAGER' ou 'ASSASSIN'
);
alter table player_roles enable row level security;
-- Note que não criamos NENHUMA política de "SELECT" para player_roles.
-- Isso significa que ninguém pode fazer consultas diretas via cliente JavaScript!

Consultando Dados Seguros via RPC (Remote Procedure Call)

Como o jogador descobre seu próprio papel se ele não tem permissão para ler de player_roles? Através de funções do Postgres escritas em PL/pgSQL.

Quando o jogador entra no jogo, o backend insere um UUID aleatório chamado de Secret Token na tabela privada player_secrets e salva esse token no localStorage do navegador do jogador. Esse token serve como assinatura secreta única do jogador.

SQL PL/pgSQL get_my_role 
create or replace function get_my_role(p_player_id uuid, p_secret_token uuid)
returns text as $$
declare
  v_role text;
begin
  -- 1. Verifica se o token secreto de fato confere com o id do jogador
  if not exists (
    select 1 from player_secrets 
    where player_id = p_player_id and secret_token = p_secret_token
  ) then
    return null;
  end if;

  -- 2. Retorna apenas o papel daquele jogador específico
  select role into v_role from player_roles where player_id = p_player_id;
  return v_role;
end;
$$ language plpgsql security definer;
💡 Conceito Chave: security definer
No Postgres, por padrão, funções rodam com os privilégios do usuário que as chamou (invoker). Como o cliente JavaScript não tem privilégios na tabela privada, a chamada falharia. Ao declarar a função como security definer, ela é executada com os privilégios do usuário dono da função (o administrador do banco), contornando as restrições normais com segurança controlada!

Garantindo o Anonimato da Missão

No Resistance, os sabotadores votam "Fracasso" de forma ultra-secreta. Se pudéssemos rastrear a ordem dos votos inseridos, saberíamos quem votou contra. A função reveal_mission_votes_secure resolve isso:

  1. Os votos individuais são inseridos em uma tabela privada temporária (mission_votes_secure).
  2. Quando a TV chama o revelador, o Postgres seleciona os votos usando ORDER BY random() para embaralhar completamente o array antes de gravá-los na tabela pública rooms.
  3. Em seguida, a função apaga os dados da tabela privada para que ninguém nunca saiba qual jogador enviou qual voto.

⚡ Sincronização em Tempo Real

O Crazy Hotel dispensa rotinas periódicas de polling (onde o app fica chamando o servidor a cada 2 segundos perguntando "o estado mudou?"). Em vez disso, ele usa o Supabase Realtime, que escuta eventos de banco via WebSocket.

No arquivo GameContext.tsx, há um hook useEffect que cria canais de escuta automática (channels):

TypeScript GameContext.tsx 
// 1. Ouvir alterações em tempo real nas salas
const roomChannel = supabase
  .channel(`room_changes:${room.id}`)
  .on(
    'postgres_changes',
    { event: 'UPDATE', schema: 'public', table: 'rooms', filter: `id=eq.${room.id}` },
    async (payload) => {
      const updatedRoom = payload.new as Room;
      setRoom(updatedRoom); // Atualiza o estado React instantaneamente!
    }
  )
  .subscribe();

O Fluxo Bidirecional

Quando um jogador no celular executa uma ação (por exemplo, vota "Aprovar Time"), o React faz um UPDATE simples na tabela players do Supabase.

O Supabase detecta a gravação no Postgres e avisa via WebSocket todos os clientes que estão escutando alterações para aquela sala. O hook no frontend da TV e dos outros Jogadores reage à mensagem atualizando seus estados internos do React, redesenhando a interface no mesmo milissegundo.

🧠 React Context como uma Máquina de Estados

O arquivo GameContext.tsx é o cérebro da aplicação no cliente. Em vez de passar variáveis e funções por vários componentes usando propriedades (prática chamada de prop-drilling), usamos a API de Contexto do React para expor dados a todo o app.

O GameContext atua como uma Máquina de Estados Finita. Ele gerencia transições limpas baseadas na coluna rooms.state:

Fases do State Machine

  • LOBBY: Aguardando os 5 a 10 jogadores entrarem.
  • ROLE_ASSIGNMENT: Papéis distribuídos. Apenas os celulares mostram o papel.
  • TEAM_SELECTION: O líder escolhe membros para ir à missão.
  • TEAM_VOTE: Todos votam se aceitam ou rejeitam o time escolhido.
  • MISSION_VOTE: Integrantes do time votam SUCESSO / FRACASSO em segredo.
  • MISSION_REVEAL: A TV revela os votos de forma embaralhada.
  • ASSASSIN_CHOICE: Se os hóspedes vencem 3 vezes, o Assassino pode tentar atirar no Gerente.
  • GAME_OVER: Exibe vencedor e papéis.

Ações Centralizadas

Toda a lógica está encapsulada em funções assíncronas simples dentro do Provider, facilitando o desenvolvimento das telas visuais. Funções expostas incluem:

  • createRoom()
  • joinRoom()
  • startGame()

🛣️ Roteamento Tipado com TanStack Router

Muitos desenvolvedores juniores usam o clássico react-router-dom. No entanto, o Crazy Hotel optou por utilizar o moderno TanStack Router.

Qual a vantagem?

O TanStack Router garante segurança de tipos (type safety) nas rotas em tempo de compilação. Se você tentar redirecionar o usuário para uma rota que não existe ou esquecer de passar parâmetros obrigatórios (como o roomCode), o TypeScript emitirá um erro antes de você subir o código para produção!

TypeScript routes.tsx 
// Definição de rotas do projeto
const rootRoute = createRootRoute({
  component: () => (
    <div className="app-container">
      <Outlet />
    </div>
  ),
});

const hostRoute = createRoute({
  getParentRoute: () => rootRoute,
  path: '/host/$roomCode', // $roomCode é um parâmetro dinâmico
  component: HostRoom,
});

const playRoute = createRoute({
  getParentRoute: () => rootRoute,
  path: '/play/$roomCode',
  component: PlayRoom,
});

Para obter o código da sala de forma segura dentro da página de jogo, basta usar o hook tipado:

TypeScript play.$roomCode.tsx 
const { roomCode } = useParams({ from: '/play/$roomCode' });

🚀 Dicas para Crescimento Profissional (Júnior a Pleno)

Se você quer se destacar no mercado, analisar este projeto traz aprendizados incríveis. Aqui estão três padrões avançados usados no Crazy Hotel para você estudar e citar em entrevistas de emprego:

1. Keepalive Beacon API

Quando um jogador fecha a aba do navegador no meio do jogo, como o banco de dados sabe que ele saiu se a conexão React foi destruída?

O Crazy Hotel resolve isso com o evento beforeunload e um fetch especial usando a flag keepalive: true. Isso garante que a requisição de desconexão seja enviada pelo navegador em segundo plano, mesmo que a página já tenha sido fechada!

2. Lógica Baseada em Transação (Atomicidade)

Evitar condições de corrida (quando duas requisições tentam alterar o mesmo dado ao mesmo tempo) é um diferencial técnico.

Em vez de puxar os dados do banco, somá-los no React e salvar de volta, o código usa a RPC append_mission_vote para executar a gravação do voto de forma atômica no Postgres. Isso garante integridade absoluta das contagens!
